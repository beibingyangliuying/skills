# AGENTS

适用于本仓库内的 Python 代码生成、修改、重构与测试任务。

本文件定义仓库级全局规则；当任务涉及特定领域时，应额外遵循对应的 `SKILL.md`。各个 skill 应优先补充领域差异与专项边界，而不是重复这里已经定义的通用 Python 规范。

## 规范等级说明

- MUST: 必须遵守的规则。
- SHOULD: 默认应遵守的规则，除非有明确理由偏离。
- MAY: 可选实践，可按任务需要采用。

## 适用范围

- MUST: 本文件适用于本仓库内的所有 Python 相关任务。
- MUST: 若某项任务命中特定 skill 的适用场景，应同时遵循对应 skill 中的专项规范。
- MUST: 若更高优先级的系统指令、开发者指令或用户指令与本文件冲突，以更高优先级指令为准。
- SHOULD: skill 只补充领域特有规则、触发边界与验证重点，不重复仓库级的通用命名、类型、测试和汇报要求。

## 全局工作原则

- MUST: 仅修改与当前任务直接相关的文件与代码。
- MUST: 不要顺手进行无关的风格清洗、批量重命名、目录重组或大规模重构，除非任务明确要求。
- MUST: 新代码优先遵循目标文件及相邻模块的既有风格。
- MUST: 接口、函数、类、变量与参数命名必须自解释；允许使用公认缩写，但禁止语义模糊的过度缩写。
- MUST: 模块内引用使用绝对导入，禁止相对导入。
- SHOULD: 新增项目内模块时尽量控制嵌套层级，避免引入过深的内部包路径；若现有结构需要更深层级，不强行重构。
- SHOULD: 优先做小步、可验证、低副作用的修改。
- SHOULD: 优先保持已有公开接口稳定；若必须改变接口，应同步更新文档与调用处。
- MAY: 在不增加维护负担的前提下，对局部代码做小范围清理，但前提是不偏离当前任务。

## 文档与类型注解

- MUST: 当参数或返回值涉及复杂结构时，文档必须说明结构约束。
- MUST: 复杂结构包括但不限于：DataFrame 的列名/索引要求、嵌套字典 Key 约束、元组各位置含义、返回对象字段约定。
- MUST: 使用 `|` 代替 `Optional` 和 `Union`。
- MUST: 禁止使用 `typing` 模块中已弃用的别名。
- MUST: 公共函数与公共类的方法应尽量提供清晰的签名与必要文档。
- SHOULD: 新增公共函数时显式标注返回类型。
- SHOULD: 一般不为函数内部简单局部变量添加类型注解；仅在有助于澄清复杂结构或避免歧义时添加。
- SHOULD: 函数文档采用 Google Style，并保持简明。
- SHOULD: 定义新函数或类时补充必要文档；更新函数或类行为时同步更新文档。
- SHOULD: 文件开头不添加 `from __future__ import annotations`，除非任务明确要求或现有文件已经统一采用。
- MAY: 对局部、一次性、语义极其明确的内部辅助函数，适度简化注解。

## 工程与实现约束

- MUST: 默认使用 `pytest` 作为测试框架。
- MUST: 若修改影响公开函数、核心逻辑、数据处理流程或 bug 修复，应优先运行相关测试；若未运行测试，必须明确说明。
- MUST: 若无法成功进入仓库规定的 Python 环境或无法执行关键命令，必须明确告知用户。
- SHOULD: 若修改影响公开接口、使用方式、依赖、命令行行为或项目说明，应同步更新 `README.md`。
- SHOULD: 仅当仓库级规范本身发生变化，或需要澄清与专项 skill 的职责分工时，才更新 `AGENTS.md`。

## 结果汇报要求

- MUST: 完成修改后，说明实际改动内容与影响范围。
- MUST: 说明是否运行测试；若未运行，说明原因。
- SHOULD: 对潜在风险、未覆盖边界或尚未验证的假设做简要说明。
