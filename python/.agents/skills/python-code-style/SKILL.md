# Python Code Style

## 何时使用

当任务包含以下任一情况时使用本 skill：

- 新增或重构 Python 函数
- 修改类定义、方法签名或模块接口
- 抽取公共逻辑
- 设计轻量数据模型或小型领域对象

## 核心目标

在遵循仓库既有风格的前提下，生成可读、可维护、抽象尺度合适的 Python 代码。

## 边界说明

- MUST: 本 skill 聚焦接口设计、抽象尺度与轻量数据模型取舍，不重复 `AGENTS.md` 中已定义的通用命名、类型注解、文档字符串与测试基线要求。
- SHOULD: 若任务具有明显的数据处理、绘图、发布兼容性或仓库工作流属性，优先同时遵循对应专项 skill。

## 执行规则

### 1. 函数与接口设计

- MUST: 函数名、参数名、返回值语义必须清晰。
- MUST: 公共函数的参数设计应稳定、直接，避免无必要的位置参数陷阱。
- SHOULD: 优先写短函数；单个函数应聚焦单一职责。
- SHOULD: 公共逻辑应提取为具名函数，而不是堆叠在调用处。
- SHOULD: 对复杂输入约束、返回结构、边界条件给出必要文档。
- MAY: 对只在局部使用且逻辑极小的辅助函数保持简洁。

### 2. 抽象尺度

- SHOULD: 优先采用直接、低认知负担的实现，不为小问题引入多余层级。
- SHOULD: 仅在提取后能明显提升复用性、可测试性或可读性时，再新增公共模块、基类或抽象层。
- SHOULD: 若抽象会掩盖实际业务语义、增加跨文件跳转成本或制造过早通用化，应保留更直接的实现。

### 3. 数据模型

- SHOULD: 若仓库中的轻量数据模型主要采用 `attrs`，则新增同类模型时优先沿用 `attrs`。
- SHOULD: 使用轻量数据模型时优先考虑不可变设计，但前提是不妨碍当前任务。
- MUST: 不要为了使用某种数据模型工具而破坏目标文件的既有范式；若目标文件已明确采用其他模式，应优先保持一致。

## 交付检查清单

在完成代码前，检查：

- 接口命名是否清晰
- 抽象层级是否与问题规模匹配
- 是否只提取了真正可复用的公共逻辑
- 轻量数据模型是否保持了与目标文件一致的范式
