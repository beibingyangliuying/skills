# Python Test Organization

## 核心目标

让测试目录结构天然表达模块上下文，使测试文件名保持简短、稳定、易发现，并降低后续重构时批量改名的成本。

## 边界说明

- MUST: 本 skill 负责测试目录结构、测试文件命名与测试组织粒度，不替代仓库级通用测试基线。
- MUST: 若任务同时涉及环境确认、测试命令选择或是否由 AI 代跑测试，优先同时遵循 `repo-workflow`。
- MUST NOT: 不要把本 skill 扩张为虚拟环境、`pytest` 入口、依赖安装或测试执行策略规范。
- SHOULD: 若仓库已有明确测试目录约定，优先在该约定内应用本 skill 的命名与组织原则，而不是强行改造仓库结构。

## 主流程

### 1. 默认镜像源码目录结构

- MUST: 默认在 `tests/` 下镜像源码目录结构组织测试，而不是把多级模块路径压进单个文件名。
- MUST: 对 `my_package/a/b/c.py`，默认优先组织为 `tests/a/b/test_c.py` 一类镜像路径。
- MUST: 设计测试路径时优先保证重构稳定性、可发现性，以及按子目录局部运行测试的便利性。

### 2. 保持测试文件名简短

- MUST: 在镜像结构下优先使用简短文件名，如 `test_<module>.py`。
- MUST NOT: 不要默认使用 `test_a_b_c.py` 这类把完整模块层级串进文件名的命名方式，除非仓库明确要求扁平测试目录且必须避免重名。
- SHOULD: 只有在镜像目录无法表达上下文时，才把少量模块信息补进文件名。

### 3. 仅在扁平目录约定下采用折中命名

- SHOULD: 若仓库测试目录必须扁平化，优先保留最近一级父模块与目标模块，例如 `test_b_c.py`，不要默认写成完整族谱式命名。
- SHOULD: 扁平目录命名的目标是解决冲突并保留足够上下文，而不是精确还原整条源码路径。

### 4. 按业务主体命名作为例外

- SHOULD: 若测试关注稳定的业务主体、核心类或功能语义，而非物理文件位置，可按业务主体命名，如 `test_order_processor.py`。
- MUST: 仅当这类命名比模块镜像更稳定、更清晰时才采用；若只是为了回避镜像目录，不要滥用业务命名。

### 5. 与测试执行规则解耦

- SHOULD: 当需要同时说明“测试放哪里”和“测试怎么跑”时，由本 skill 负责目录结构与命名，由 `repo-workflow` 负责环境确认、命令入口与执行边界。

## 交付检查清单

在完成任务前，检查：

- 是否优先采用了 `tests/` 下的镜像目录结构
- 测试文件名是否保持简短，并让目录而不是文件名承担主要上下文信息
- 是否避免默认使用 `test_a_b_c.py` 一类冗长命名
- 若采用扁平目录，是否只保留了解决冲突所需的最小父级上下文
- 若采用业务主体命名，是否确实比模块镜像更稳定、更清晰
- 若任务还涉及测试命令与执行方式，是否把对应部分交给了 `repo-workflow`
