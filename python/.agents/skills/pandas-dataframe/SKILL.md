# Pandas DataFrame

## 何时使用

当任务包含以下内容时使用本 skill：

- 编写或重构 pandas 数据处理代码
- 新增 DataFrame/Series 处理函数
- 修改列名、索引、分组、聚合、透视、合并逻辑
- 整理复杂链式操作
- 对数据处理函数补充输入输出约束说明

## 核心目标

写出可读、可验证、约束清晰的数据处理代码，避免隐式假设与脆弱转换。

## 边界说明

- MUST: 本 skill 只聚焦 DataFrame/Series 数据处理行为，不替代绘图、仓库环境或发布兼容性规范。
- SHOULD: 若任务重点是出图与导出，优先使用 `matplotlib-figures`；若重点是通用 Python 接口整理，优先遵循 `AGENTS.md` 与其他通用 skill。

## 执行规则

### 1. 结构约束必须显式

- MUST: 若函数输入或输出为 DataFrame/Series，文档中必须说明关键结构约束。
- MUST: 至少在必要时说明以下内容：
  - 必需列名
  - 索引含义
  - 分组键
  - 结果行粒度
  - 关键缺失值假设
- SHOULD: 对“输入必须已排序”“列必须唯一”“索引必须单调”等前置条件做显式说明。

### 2. 写法选择

- SHOULD: 在可读性良好的前提下优先采用链式操作。
- SHOULD: 若链过长、难调试或难复用，应拆成命名清晰的中间步骤。
- SHOULD: 中间变量应有语义，不要使用 `tmp`, `df1`, `df2` 这类在复杂处理中难以理解的名称，除非作用域极小。
- MUST: 不要为了追求“链式”而牺牲可读性。

### 3. 数据安全与稳健性

- MUST: 对可能依赖列存在性的逻辑保持谨慎，不默认假设列一定存在，除非上游约束已明确。
- SHOULD: 对关键列名、分组键、合并键保持一致命名。
- SHOULD: 对易出错的 reshape / merge / pivot / explode 操作，优先保留清晰的步骤边界。
- SHOULD: 对可能改变行数、粒度或唯一性的操作，在代码结构或文档中体现出来。

### 4. 结果表达

- SHOULD: 数据处理函数尽量返回结构稳定、列名清晰的结果。
- SHOULD: 避免返回“靠约定猜测含义”的匿名元组或字段不稳定的对象。
- MUST: 若返回 DataFrame 的列具有业务含义，文档必须写清楚。
- MAY: 对结果列进行适度排序或重排列顺序，以增强可读性，但不要破坏既有接口约定。

## 交付检查清单

在完成代码前，检查：

- 文档是否写明所需列与结果结构
- 链式操作是否仍然可读
- 是否存在隐式依赖列名或索引的逻辑
- 结果粒度是否清晰
- 是否引入了不必要的中间副作用
