# Verification Scope

## 目标

以最小但可信的方式验证修改结果，并对未执行的验证保持诚实。

## 测试执行

- MUST: 环境确认完成后，默认自行运行最相关的最小测试集，以最小但可信的方式验证修改结果。
- MUST: 若测试成本、执行风险、运行耗时或当前环境能力使立即执行不合适，必须明确说明未执行原因、建议补跑的命令，以及当前验证缺口。
- SHOULD: 默认推荐使用 `pytest` 作为测试入口。
- SHOULD: 修改公开函数、核心逻辑、数据处理流程或 bug 修复后，优先运行最相关的最小测试集。
- SHOULD: 能跑单测时优先推荐单测，避免无必要地默认扩大为全量测试。
- SHOULD: 若需要为用户补充命令示例，优先提供可直接执行的 PowerShell 命令。
- MUST: 若 AI 未实际运行测试，必须明确说明未执行原因、本次建议补跑的测试命令、哪些测试仍待执行，以及当前验证缺口是什么。
- MUST: 若 AI 实际运行了测试，必须明确区分哪些结果来自 AI 本次执行，哪些验证仍待用户完成。
- MAY: 为保持测试简洁，在非常小的纯函数中补充 doctest，但不替代主要测试策略。

## 脚本与命令验证

- SHOULD: 先做最小可验证修改，再运行最小必要命令验证。
- SHOULD: 若任务仅涉及静态改写且无法安全运行命令，应明确说明未执行命令验证。
- MUST: 不要假装命令已经成功运行；只报告实际执行和实际结果。
- MUST: 不得把尚未由用户执行的测试命令描述成已验证通过。

## 使用建议

- 优先选最相关的验证，而不是默认扩大为全量检查。
- 当环境、权限或成本使验证不可行时，在结果中显式写出缺口。
