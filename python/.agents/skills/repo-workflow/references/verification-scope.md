# Verification Scope

## 目标

以最小但可信的方式验证修改结果，并对未执行的验证保持诚实。

## 测试执行

- MUST: 凡是测试任务，必须先询问用户是否现在执行测试。
- MUST: 默认推荐使用 `pytest`，但测试由用户执行，AI 不直接运行测试命令。
- SHOULD: 修改公开函数、核心逻辑、数据处理流程或 bug 修复后，优先向用户提供最相关的最小测试集对应的 PowerShell 命令。
- SHOULD: 能跑单测时优先给用户单测命令，避免无必要地默认扩大为全量测试。
- MUST: 若用户同意执行测试，AI 必须先提供可直接复制执行的 PowerShell 命令，并等待用户回传测试输出。
- MUST: 若未运行测试，必须明确说明原因；“等待用户执行并回传结果”属于标准原因之一。
- MAY: 为保持测试简洁，在非常小的纯函数中补充 doctest，但不替代主要测试策略。

## 脚本与命令验证

- SHOULD: 先做最小可验证修改，再运行最小必要命令验证。
- SHOULD: 若任务仅涉及静态改写且无法安全运行命令，应明确说明未执行命令验证。
- MUST: 不要假装命令已经成功运行；只报告实际执行和实际结果。

## 使用建议

- 优先选最相关的验证，而不是默认扩大为全量检查。
- 当环境、权限或成本使验证不可行时，在结果中显式写出缺口。
