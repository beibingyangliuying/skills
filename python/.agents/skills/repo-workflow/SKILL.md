# Repo Workflow

## 何时使用

当任务包含以下内容时使用本 skill：

- 运行 Python 命令
- 执行测试
- 运行脚本
- 确认虚拟环境
- 修改脚本型 Python 文件

## 核心目标

在正确的仓库环境中执行命令，并以最小代价验证修改结果。

## 执行规则

### 1. 环境激活

- MUST: 在运行本仓库相关 Python 命令前，先尝试激活仓库规定的虚拟环境。
- MUST: 尝试使用以下命令激活虚拟环境： `& <Project>/.venv/Scripts/Activate.ps1`。
- MUST: 若虚拟环境已成功激活，后续所有以 `uv` 开头的命令都必须显式追加 `--active`，包括 `uv run`、`uv pip`、`uv sync` 等常见形式。
- SHOULD: 将 `--active` 视为已激活环境下 `uv` 命令的默认写法，以明确使用当前激活环境，避免落到错误解释器或环境解析逻辑。
- MUST: 若无法成功激活环境，必须明确告知用户。
- SHOULD: 若一个路径失败，再尝试另一路径。
- SHOULD: 仅在确认环境可用后再执行依赖该环境的 Python 命令。

### 2. 测试执行

- MUST: 默认使用 `pytest`。
- SHOULD: 修改公开函数、核心逻辑、数据处理流程或 bug 修复后，优先运行最相关的最小测试集。
- SHOULD: 能跑单测时优先跑单测，避免无必要的全量测试。
- MUST: 若未运行测试，必须明确说明原因。
- MAY: 为保持测试简洁，在非常小的纯函数中补充 doctest，但不替代主要测试策略。

### 3. 脚本文件约定

- SHOULD: 脚本型 Python 文件保持 notebook 风格注释，以便分单元执行。
- MUST: 若目标文件已经采用另一种清晰的脚本组织方式，则优先保持原风格。
- SHOULD: 对步骤性脚本保留清晰段落与注释边界，便于复现。

### 4. 命令与验证

- SHOULD: 先做最小可验证修改，再运行最小必要命令验证。
- MUST: 若已成功激活虚拟环境，则后续实际执行的所有 `uv` 命令都必须包含 `--active`。
- SHOULD: 若任务仅涉及静态改写且无法安全运行命令，应明确说明未执行命令验证。
- MUST: 不要假装命令已经成功运行；只报告实际执行和实际结果。
- MUST: 汇报 `uv` 命令执行结果时，必须基于实际执行过且包含 `--active` 的命令；若环境未成功激活，不得假设 `uv` 已使用正确环境。

## 交付检查清单

在完成任务前，检查：

- 是否已尝试激活正确环境
- 若已激活环境，是否为所有实际执行的 `uv` 命令附加了 `--active`
- 是否运行了相关 `pytest`
- 是否如实汇报了未执行测试的原因
- 脚本文件是否保持了可读的执行分段
