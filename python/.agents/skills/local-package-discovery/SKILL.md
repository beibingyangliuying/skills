# Local Package Discovery

## 核心目标

统一识别当前项目根目录中的本地包发布产物，并为后续复用、升级或适配决策提供一致的候选列表与选择依据。

## 执行规则

### 1. 识别范围

- MUST: 将“本地包”限定为当前项目根目录下、从其他项目拷入的 `.whl` 或 `.tar.gz` 发布产物。
- MUST: 不要把当前项目自己的 `dist/`、构建缓存或其他发布流程产物误判为这里的“本地包”。
- MUST: 若根目录不存在 `.whl` 或 `.tar.gz`，明确说明“未发现本地包发布产物”，不要虚构候选。

### 2. 候选发现与归组

- MUST: 先扫描当前项目根目录中的 `.whl` 与 `.tar.gz` 文件，再开始后续设计或实现判断。
- MUST: 将同一分发名的文件视为一组候选。
- MUST: 默认选择版本最高的候选作为该分发名的最新版候选。
- SHOULD: 优先从文件名推断分发名、版本、可能的导入名与能力定位。
- SHOULD: 若文件名信息不足，再结合可读元数据、仓库中的 import 线索与调用语义补充判断。

### 3. 不确定性处理

- MUST: 若无法从文件名、元数据或代码线索判断某个产物的用途，如实说明证据不足。
- MUST: 若放弃某个候选，不要省略原因；应明确说明是版本较旧、语义不匹配、信息不足，还是目标任务与其能力边界不符。
- SHOULD: 当多个候选都可能匹配时，优先给出“默认最新版候选 + 备选原因”的结论，而不是假装只有唯一答案。

### 4. 输出要求

- MUST: 明确列出识别到了哪些候选产物。
- MUST: 明确说明每个分发名最终选择了哪个最新版候选。
- MUST: 明确说明选择依据，例如版本排序、文件名线索、可读元数据或仓库调用语义。
- MUST: 若未采用某个候选，明确说明未采用原因。

## 交付检查清单

在完成任务前，检查：

- 是否已扫描当前项目根目录中的 `.whl` / `.tar.gz`
- 是否已按分发名归组并选出最新版候选
- 是否已区分本地包与当前项目自己的 `dist/` 或构建产物
- 是否已如实说明证据不足或未采用原因
- 是否已输出候选列表、最终选择与依据
