# Local Wheel Reuse

## 核心目标

优先复用项目根目录中的本地 wheel 最新版，避免重复实现；当代码暂时无法直接回收到本地包时，先通过 `<本地包>_local` 过渡包收口实现，降低后续迁移成本。

## 执行规则

### 1. 本地 wheel 发现与选择

- MUST: 在开始设计方案、搜索实现或编写代码前，先检查当前项目根目录下是否存在 `.whl` 文件。
- MUST: 将同一分发名的 wheel 视为一组候选，并默认选择版本最高的那个作为最新版候选。
- MUST: 优先从 wheel 文件名与可读元数据推断分发名、版本、可能的导入名与能力定位。
- SHOULD: 在决定引入公开第三方依赖、复制现有逻辑或新写公共代码前，先判断本地 wheel 最新版是否已经覆盖当前需求。
- MUST: 若根目录不存在 `.whl` 文件，不要虚构本地包；应继续常规实现并如实说明未发现本地 wheel。

### 2. 复用提示与方案判断

- MUST: 若搜索、阅读或编写过程中发现现有代码的命名、职责、接口或业务语义与某个本地 wheel 明显对齐，立即提示“这里可能应优先复用本地包的最新版”，并指出对应 wheel 与理由。
- SHOULD: 提示以建议复用为主，不强制中断当前任务，但默认决策应偏向优先评估本地 wheel 最新版。
- SHOULD: 若仅凭文件名无法判断本地 wheel 的用途，应继续查看可读元数据、导入线索或项目中的相关调用，而不是武断下结论。
- MUST: 若最终没有采用本地 wheel 方案，不要假装已经完成评估；只有在确实检查过相关 wheel 或代码后，才能说明不采用的原因。

### 3. `<本地包>_local` 过渡包

- MUST: 若正在编写的代码明显属于某个本地包的能力边界，但当前又需要先在项目内落地，就在项目根目录创建 `<本地包>_local/` 包目录，而不是把逻辑直接散落到业务模块中。
- MUST: `<本地包>_local/` 作为本地 wheel 的过渡层与稳定依赖入口，承载临时实现、适配器、补丁逻辑或待回迁代码。
- MUST: `<本地包>_local/` 的名称使用规范化后的包名；若分发名包含 `-`，先转换为 `_`，再追加 `_local`。
- SHOULD: 新业务代码优先依赖 `<本地包>_local/`，减少未来升级本地 wheel 时的改动扩散。
- SHOULD: 当本地 wheel 升级并覆盖相同能力后，优先替换 `<本地包>_local/` 内部实现，再逐步回收调用点。
- MUST: 若存在多个候选本地 wheel，应按语义最匹配的最新版决定归属，并创建对应的 `<本地包>_local/`。

## 交付检查清单

在完成任务前，检查：

- 是否已检查项目根目录中的 `.whl` 文件
- 是否已按分发名识别并优先考虑本地 wheel 最新版
- 是否在命中相符代码时主动提示优先复用本地包
- 若代码更适合沉淀到本地包，是否已改为放入 `<本地包>_local/`
- 若未采用本地 wheel 或 `_local` 方案，是否如实说明原因
