# Local Package Integration

## 何时使用

当任务包含以下内容时使用本 skill：

- 评估是否应优先复用当前项目附近的本地包能力
- 将现有代码改为依赖本地包或围绕本地包建立适配层
- 设计 `<本地包>_local` 过渡包来收口临时实现、适配器或补丁逻辑

## 核心目标

优先复用本地包最新版候选，避免重复实现；当代码暂时无法直接回收到本地包时，以低扩散、可迁移的方式组织过渡层。

## 边界说明

- MUST: 在本 skill 中，本地包候选识别应先遵循 `local-package-discovery` 的规则。
- MUST: 本 skill 面向“接入与组织方式”决策，不专门处理版本升级带来的兼容性改造。
- SHOULD: 若任务重点是识别候选而非接入方案，优先使用 `local-package-discovery`。
- SHOULD: 若任务重点是将旧版本升级到新版本并梳理 breaking changes，优先使用 `local-package-upgrade-compatibility`。

## 执行规则

### 1. 本地包识别前置

- MUST: 在开始设计方案、搜索实现或编写代码前，先按 `local-package-discovery` 的规则检查默认扫描路径下是否存在本地 `.whl` 或 `.tar.gz` 发布产物。
- MUST: 将 discovery 阶段选出的最新版候选视为默认复用入口，而不是重新各自发明一套识别规则。
- SHOULD: 在决定引入公开第三方依赖、复制现有逻辑或新写公共代码前，先判断本地包最新版是否已经覆盖当前需求。
- MUST: 若 discovery 阶段未发现可用本地包，不要虚构本地包；应继续常规实现并如实说明未发现候选产物。

### 2. 复用提示与方案判断

- MUST: 若搜索、阅读或编写过程中发现现有代码的命名、职责、接口或业务语义与某个本地包明显对齐，及时提示“这里可能应优先复用本地包的最新版”，并指出对应产物与理由。
- SHOULD: 默认决策应偏向优先评估本地包最新版，但不强制中断所有其他实现路径。
- SHOULD: 若仅凭文件名无法判断本地包用途，应继续沿用 discovery 阶段的元数据与代码线索，而不是武断下结论。
- MUST: 若最终没有采用本地包方案，只有在确实检查过相关产物或代码后，才能说明不采用的原因。

### 3. `<本地包>_local` 过渡包

- SHOULD: 若正在编写的代码明显属于某个本地包的能力边界，但当前又需要先在项目内落地，优先评估是否用 `<本地包>_local/` 作为过渡层与稳定依赖入口。
- MUST: 仅当逻辑明确属于过渡实现、预计会被多个调用点复用、且后续确实计划回迁到本地包时，才创建 `<本地包>_local/`。
- SHOULD: 若只是一次性的局部适配或单点胶水代码，不要机械创建 `<本地包>_local/`；应优先选择更贴近调用边界的组织方式。
- MUST: 若创建 `<本地包>_local/`，名称使用规范化后的包名；若分发名包含 `-`，先转换为 `_`，再追加 `_local`。
- SHOULD: 当 `<本地包>_local/` 已建立时，新业务代码优先依赖该过渡入口，减少未来升级本地包时的改动扩散。
- SHOULD: 当本地包升级并覆盖相同能力后，优先替换 `<本地包>_local/` 内部实现，再逐步回收调用点。

## 交付检查清单

在完成任务前，检查：

- 是否已先按 `local-package-discovery` 的规则检查候选本地包
- 是否已按分发名识别并优先考虑本地包最新版
- 是否在命中相符代码时主动提示优先复用本地包
- 若采用 `<本地包>_local/`，是否满足过渡层、复用度和回迁预期这三个条件
- 若未采用本地包或 `_local` 方案，是否如实说明原因
