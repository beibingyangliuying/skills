# Dynamic Usage Risks

## 目标

在存在动态调用、注册机制或兼容承诺时保持保守，并在清理后完成最小可信验证。

## 动态使用风险

- MUST: 对装饰器注册、插件注册、CLI 子命令、反射、字符串路径引用、测试夹具与测试发现机制保持警惕。

## 清理动作

- MUST: 对已确认无用的符号、辅助模块或兼容层，默认直接删除。
- MUST: 删除主体时同步清理伴生垃圾，例如失效 import、过期注释、陈旧 docstring 与无意义包装转发。
- SHOULD: 若任务是模块级清理，优先做局部、最小化修改，避免顺手调整无关结构。
- MUST: 若代码虽然当前模块未用，但属于公共 API、导出面或兼容承诺的一部分，不要删除；应改为向用户说明其公开接口属性。

## 验证与汇报

- MUST: 清理后至少再次搜索相关引用，确认删除对象未被遗漏引用。
- SHOULD: 若可以安全运行测试或静态检查，优先执行最相关的最小集合验证结果。
- MUST: 明确汇报实际执行了哪些验证，以及哪些验证未执行。
- MUST: 明确汇报删除了什么，以及为什么判定这些代码属于死代码。
