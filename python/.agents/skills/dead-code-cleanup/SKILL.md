# Dead Code Cleanup

## 核心目标

优先删除证据充分的死代码，并在动态调用风险存在时保持保守，避免把一般性的重构请求误扩展成删除动作。

## 边界说明

- MUST: 仅在用户明确表达“删除未使用代码”或“清理死代码”意图时使用本 skill。
- MUST: 不要因为一般性的“重构”“整理代码”“优化结构”请求自动触发本 skill。
- SHOULD: 若任务重点是接口整理、兼容性保护或实现迁移，优先使用更贴合场景的其他 skill，而不是借本 skill 扩大删除范围。

## 主流程

### 1. 死代码识别范围

- SEE: 需要清理边界和证据检查清单时，读取 `references/evidence-checklist.md`。

### 2. 证据优先判定

- MUST: 在删除任何候选死代码前，先做引用搜索。
- MUST: 同时检查直接引用与导出面，例如 `__all__`、包级 re-export、入口模块转发与公共 API 暴露。
- MUST: 不要因为“当前文件内未引用”就直接判定为死代码。
- MUST: 若证据不足或存在动态调用风险，先保留代码并向用户说明风险点。

### 3. 清理动作

- MUST: 对已确认无用的符号、辅助模块或兼容层，默认直接删除。
- MUST: 若代码虽然当前模块未用，但属于公共 API、导出面或兼容承诺的一部分，不要删除；应改为向用户说明其公开接口属性。
- SEE: 需要动态调用风险、伴生清理和验证细则时，读取 `references/dynamic-usage-risks.md`。

### 4. 验证与汇报

- MUST: 清理后至少再次搜索相关引用，确认删除对象未被遗漏引用。
- MUST: 明确汇报实际执行了哪些验证，以及哪些验证未执行。
- MUST: 明确汇报删除了什么，以及为什么判定这些代码属于死代码。

## 交付检查清单

在完成任务前，检查：

- 是否确认用户明确表达了死代码清理意图
- 是否限定了任务边界为项目级或模块级
- 是否完成引用搜索与导出面检查
- 是否排除了注册、反射、入口点等动态用法
- 是否一并清理了伴生 import 和包装层
- 是否执行并如实汇报了最小相关验证
