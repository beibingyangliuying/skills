# Dead Code Cleanup

## 何时使用

当任务包含以下内容时使用本 skill：

- 用户明确要求删除未使用函数、类、方法、常量、模块或兼容层
- 用户明确要求识别并清理死代码、废弃包装层或过期分支
- 用户给出项目级或模块级的死代码清理范围

## 核心目标

优先删除证据充分的死代码，并在动态调用风险存在时保持保守，避免把一般性的重构请求误扩展成删除动作。

## 边界说明

- MUST: 仅在用户明确表达“删除未使用代码”或“清理死代码”意图时使用本 skill。
- MUST: 不要因为一般性的“重构”“整理代码”“优化结构”请求自动触发本 skill。
- SHOULD: 若任务重点是接口整理、兼容性保护或实现迁移，优先使用更贴合场景的其他 skill，而不是借本 skill 扩大删除范围。

## 执行规则

### 1. 死代码识别范围

- MUST: 默认覆盖以下 4 类死代码：未引用函数/类/方法、未使用模块级常量、无导入或无调用的辅助模块、过期兼容分支或包装层。
- MUST: 同时支持“全项目清理”和“指定模块清理”两种任务边界。
- SHOULD: 先明确当前任务是项目级还是模块级，再开始搜索引用与制定删除范围。
- SHOULD: 优先删除真正无用的代码，不顺手扩展成无关重构任务。

### 2. 证据优先判定

- MUST: 在删除任何候选死代码前，先做引用搜索。
- MUST: 同时检查直接引用与导出面，例如 `__all__`、包级 re-export、入口模块转发与公共 API 暴露。
- MUST: 对装饰器注册、插件注册、CLI 子命令、反射、字符串路径引用、测试夹具与测试发现机制保持警惕。
- MUST: 不要因为“当前文件内未引用”就直接判定为死代码。
- MUST: 若证据不足或存在动态调用风险，先保留代码并向用户说明风险点。

### 3. 清理动作

- MUST: 对已确认无用的符号、辅助模块或兼容层，默认直接删除。
- MUST: 删除主体时同步清理伴生垃圾，例如失效 import、过期注释、陈旧 docstring 与无意义包装转发。
- SHOULD: 若任务是模块级清理，优先做局部、最小化修改，避免顺手调整无关结构。
- MUST: 若代码虽然当前模块未用，但属于公共 API、导出面或兼容承诺的一部分，不要删除；应改为向用户说明其公开接口属性。

### 4. 验证与汇报

- MUST: 清理后至少再次搜索相关引用，确认删除对象未被遗漏引用。
- SHOULD: 若可以安全运行测试或静态检查，优先执行最相关的最小集合验证结果。
- MUST: 明确汇报实际执行了哪些验证，以及哪些验证未执行。
- MUST: 明确汇报删除了什么，以及为什么判定这些代码属于死代码。

## 交付检查清单

在完成任务前，检查：

- 是否确认用户明确表达了死代码清理意图
- 是否限定了任务边界为项目级或模块级
- 是否完成引用搜索与导出面检查
- 是否排除了注册、反射、入口点等动态用法
- 是否一并清理了伴生 import 和包装层
- 是否执行并如实汇报了最小相关验证
