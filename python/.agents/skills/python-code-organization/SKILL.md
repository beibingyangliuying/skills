# Python Code Organization

## 核心目标

让读者按从上到下的阅读顺序，自然理解模块与类的职责展开关系；排序依据优先是阅读路径与依赖层次，而不是机械地按“先类后函数”或“公有私有”分桶。

## 边界说明

- MUST: 本 skill 只约束模块内与类内的代码编排、职责归组和文件级 `region` 使用，不替代仓库级通用代码规范。
- MUST: 若任务重点是函数签名、抽象层次、接口设计或轻量数据模型取舍，优先同时遵循 `python-code-style`，不要让本 skill 扩张为接口设计规范。
- MUST: 若任务重点是删除未使用代码、过时代码或兼容层，优先使用 `dead-code-cleanup`，不要借“整理结构”之名扩大为删除动作。
- MUST: 不要因为整理顺序而顺手改公开接口、批量重命名、删除代码或做无关风格清洗，除非用户明确要求。

## 主流程

### 1. 先识别阅读主线

- MUST: 在重排前先识别模块中的阅读主线，例如入口函数、核心类、支撑逻辑、底层辅助函数，以及异常、常量、轻量类型等上下文定义。
- SHOULD: 先判断“读者最应该先理解什么”，再决定各定义的相对位置，而不是先按符号类别机械分组。

### 2. 按理解路径组织顶层定义

- MUST: 顶层定义默认按“必要的模块上下文 -> 核心入口或核心类 -> 与其紧邻的配套逻辑 -> 私有或低层辅助实现”的理解路径组织。
- MUST: 相近功能的类和函数应放在一起；若某个功能组天然混合类与函数，不要为了维持“类先于函数”把它们拆散。
- SHOULD: 让调用方、阅读者或后续维护者最先接触到主流程，再逐步下沉到辅助细节。
- SEE: 需要模块级排序启发式、类内排序细则与示例时，读取 `references/ordering-and-regions.md`。

### 3. 仅在有清晰分组时使用文件级 `region`

- MUST: 当模块被整理成多个清晰的功能组时，默认使用文件级 `region` 包裹各组内容。
- MUST: `region` 名称应直接体现职责，例如“参数解析”“核心流程”“序列化支持”；不要使用 `helpers`、`misc`、`others` 一类低信息量名字。
- MUST: 不要为了单个定义或极短文件机械添加 `region`，也不要把文件切成过度细碎的分段。
- MUST: 类内部不要求使用 `region`。

### 4. 按自然递进组织类内成员

- MUST: 类内成员默认按“类级常量或字段语境 -> `__init__` 与必要构造入口 -> 对外主要方法 -> 对外辅助方法 -> 私有辅助方法”的顺序组织。
- SHOULD: 需要配套理解的方法应相邻放置，而不是严格按可见性或字母顺序打散。
- SHOULD: 若某个私有辅助方法只服务于紧邻的主流程方法，可保留在其附近，以降低阅读跳转成本。

## 交付检查清单

在完成任务前，检查：

- 是否先识别了模块或类的阅读主线
- 顶层定义是否按理解路径而不是按符号类别机械堆放
- 相近功能的类和函数是否被放在一起
- 文件级 `region` 是否只在存在多个清晰功能组时使用
- `region` 名称是否直接表达职责，而不是使用空泛标签
- 类内顺序是否体现从主流程到辅助细节的自然递进
- 是否避免把结构整理扩展成删除代码、改接口或无关风格清洗
