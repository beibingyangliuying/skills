# Ordering And Regions

## 目标

把“自然递进”的判断收敛为稳定、可执行的启发式，减少主观随意排序。

## 模块级排序启发式

- MUST: 先放理解主流程所需的最小上下文，例如模块级常量、异常、轻量类型或协议约束。
- SHOULD: 核心入口函数、核心类或最值得先阅读的主流程定义，应尽量提前出现。
- SHOULD: 与核心入口强相关的配套函数、配套数据结构或适配逻辑，应靠近核心入口放置。
- SHOULD: 私有工具函数、低层转换逻辑和通用辅助实现，默认后置，除非提前出现能明显降低理解成本。
- MUST: 不要单纯为了“所有类在前、所有函数在后”而拆散同一职责链路上的定义。
- MUST: 不要为了形式整齐把调用关系紧密的代码分散到相距很远的位置。

## 类内排序启发式

- MUST: 默认先建立类的语境，再展开行为。优先级依次为：类级常量与字段语境、`__init__`、替代构造入口、对外主要方法、对外辅助方法、私有辅助方法。
- SHOULD: 若某个公开方法是该类最主要的使用入口，应尽量靠前。
- SHOULD: 一组协同工作的公开方法可以相邻放置，不必被零散辅助方法打断。
- SHOULD: 只服务于某个主方法的私有辅助方法，可以放在该主方法之后，哪怕它不是文件中最后一个私有方法。
- MUST: 不要按字母顺序重排类方法，除非目标文件本来就明确采用该模式，且这样做不会破坏主流程可读性。

## 文件级 `region` 使用规则

- MUST: 只有在模块存在两个及以上清晰功能组时，才默认新增文件级 `region`。
- SHOULD: 每个 `region` 应覆盖一个完整职责片段，而不是只包裹一个零散函数或一个偶然相邻的定义。
- SHOULD: `region` 划分应服务于阅读导航；若加上后比不加更碎、更绕或更像目录噪声，就不要加。
- MUST: `region` 名称使用职责语义，不使用 `misc`、`helpers`、`utils`、`other` 之类泛化标签，除非这些词本身就是该组公开语义的一部分。
- MUST: 类内部不要求使用 `region`；类内主要靠成员顺序表达自然递进。

## 正反例

### 正例

- SHOULD: 在一个命令行模块中，先放参数对象或异常，再放 `main()` 与主执行流程，最后放解析细节与字符串格式化辅助函数。
- SHOULD: 在一个序列化模块中，先放核心数据类，再放与该数据类紧邻的读写入口，最后放底层字段映射与校验辅助函数。
- SHOULD: 在一个服务类中，先放 `__init__` 和最常用的公开入口方法，再放支撑这些入口的方法，最后放私有拼装或转换逻辑。

### 反例

- MUST NOT: 因为“所有工具函数都应后置”，就把核心类依赖理解的关键转换函数挪到文件极末尾，导致主流程阅读时来回跳转。
- MUST NOT: 因为“类优先于函数”，把同一功能组中的工厂函数和核心类拆到两个相距很远的区域。
- MUST NOT: 给一个只有三个定义的小模块强行补上多个 `region`，让结构噪声大于导航收益。
- MUST NOT: 为了结构看起来整齐，把类的主要公开方法埋在大量次要辅助方法之后。
