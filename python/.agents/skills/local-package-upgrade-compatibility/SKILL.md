# Local Package Upgrade Compatibility

## 核心目标

指导 AI 在升级本地包到新版本时，系统识别受影响边界、实施必要的兼容性改造，并在允许 breaking change 的前提下清楚说明影响与迁移方式。

## 执行规则

### 1. 升级前识别

- MUST: 先按 `local-package-discovery` 的规则识别当前项目根目录中的本地 `.whl` / `.tar.gz` 候选。
- MUST: 明确区分升级前正在使用的旧版本候选与准备对齐的新版本候选。
- MUST: 若未找到足够证据判断目标版本，不要假装升级目标已明确；应说明缺失的是文件、版本还是调用线索。

### 2. 受影响边界盘点

- MUST: 在修改前先盘点受影响边界，包括 import 路径、导出符号、函数签名、返回结构、异常语义、配置、CLI、序列化字段与适配层。
- MUST: 优先从仓库中的实际调用点、适配代码、文档示例与测试覆盖判断影响面，而不是只看包名猜测。
- SHOULD: 将“需要直接替换的调用点”和“可以通过局部适配吸收的差异”分开描述。

### 3. 升级策略

- MUST: 以对齐新版本接口与行为为优先目标。
- MUST: 允许为了对齐新版本而接受 breaking change，不要求默认保留旧接口兼容层。
- MUST: 若决定保留兼容层、别名或过渡适配，明确说明保留范围与退出条件。
- SHOULD: 仅在兼容层能显著降低迁移成本且不会掩盖真实升级语义时保留过渡层。

### 4. 结果汇报

- MUST: 明确说明升级前后涉及的包名与版本。
- MUST: 明确列出识别到的 breaking changes。
- MUST: 明确说明当前项目中受影响的调用点或边界类型。
- MUST: 明确说明需要的适配、替换或删除动作。
- MUST: 提供推荐迁移路径与最小验证项。

### 5. 与其他 skill 的边界

- MUST: 将本 skill 视为“依赖本地包升级带来的改造”规则，不要把它与“当前项目本身作为发布包的向后兼容”混为一谈。
- SHOULD: 若任务重点是复用本地包现有能力而不是升级版本，优先使用 `local-package-reuse`。
- SHOULD: 若任务重点是保护当前仓库作为对外发布包的兼容性，优先使用 `package-compatibility`。

## 交付检查清单

在完成任务前，检查：

- 是否已识别旧版与新版本地包候选
- 是否已盘点 import、导出、签名、返回结构、异常、配置、CLI 与序列化字段影响
- 是否已明确哪些 breaking change 是为了对齐新版本必须接受的
- 是否已说明需要的适配、替换或删除动作
- 是否已提供迁移路径与最小验证项
