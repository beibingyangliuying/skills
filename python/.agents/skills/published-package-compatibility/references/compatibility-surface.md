# Compatibility Surface

## 目标

在确认当前仓库属于对外发布包后，默认把用户升级体验放在内部整洁性之前。

## 兼容性优先级

- MUST: 默认优先保持公共 import 路径稳定。
- MUST: 默认优先保持模块导出、函数签名、参数语义、返回结构与异常行为稳定。
- MUST: 默认优先保持 CLI 参数、子命令、退出码语义与帮助信息中的关键用法稳定。
- MUST: 默认优先保持配置 key、配置文件结构、序列化字段、文件格式与默认行为稳定。
- MUST: 不要在未明确告知用户的情况下引入破坏性重命名、删除、重排、收紧类型约束或改变默认值语义。
- SHOULD: 将“用户现有代码是否还能运行”视为高于“内部实现是否更整洁”的约束。
- SHOULD: 若保留旧 API、旧别名或兼容层，也要把“是否明确提示迁移”视为兼容性的一部分，而不是只保证还能运行。

## 实施策略

- SHOULD: 优先做增量兼容修改，而不是直接替换旧接口。
- SHOULD: 若必须重构，优先保留旧别名、兼容层、re-export、参数兜底或带弃用 warning 的过渡入口。
- MUST: 若保留旧 API、旧别名或兼容层，不要只做静默兼容；旧调用路径应伴随弃用 warning，明确提示迁移。
- SHOULD: 优先通过 `deprecated` 包实现弃用提示；若不适合引入该方案，至少保留等价 warning 机制。
- SHOULD: 修改模块布局或 `src/` 包结构前，先检查公开导入路径是否会失效。
- SHOULD: 调整函数参数时，优先新增可选参数，而不是直接删除、改名或重排现有参数。
- SHOULD: 调整返回值时，优先保持旧结构可用；若必须扩展，尽量采用向后兼容方式。
- SHOULD: 升级依赖时，同时检查 Python 版本下限、第三方依赖版本范围、安装方式与用户环境影响。
- MAY: 当旧行为确实需要淘汰时，先引入弃用 warning，再在明确版本边界下移除。
