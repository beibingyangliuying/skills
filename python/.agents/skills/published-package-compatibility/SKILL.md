# Published Package Compatibility

## 何时使用

当任务包含以下内容时使用本 skill：

- 修改当前仓库对外暴露的 import 路径、导出面、函数签名或返回结构
- 调整当前仓库的 CLI、配置格式、打包元数据、入口点或目录结构
- 在发布前重构当前仓库代码，并需要评估对现有用户的兼容性影响

## 核心目标

在识别到当前仓库很可能是对外发布的 Python 包时，默认优先保护现有用户的升级体验，尽量避免无说明的破坏性变更。

## 边界说明

- MUST: 本 skill 限定为“当前仓库本身作为对外发布包”的兼容性保护规则。
- MUST: 不要把“识别根目录本地包发布产物”的逻辑重新塞回本 skill。
- SHOULD: 若任务重点是升级项目附近的本地 `.whl` / `.tar.gz` 并改造适配，优先使用 `local-package-upgrade-compatibility`。
- SHOULD: 若任务重点只是评估或接入本地包能力，优先使用 `local-package-discovery` 或 `local-package-integration`。

## 执行规则

### 1. 识别规则

- MUST: 在修改公开接口、目录结构、依赖、CLI、配置或文件格式前，先判断当前项目是否属于对外发布的包。
- MUST: 不要仅因根目录存在 `dist/` 或单个构建产物，就直接认定当前项目是对外发布包。
- MUST: 将以下情况视为强信号：
  - 根目录存在 `pyproject.toml`，且包含标准包元数据、入口点或 build backend
  - 存在 `src/<package_name>/` 等清晰的发布包结构，并能与项目名、文档或入口点对应
  - README、示例代码、测试或公开文档中存在面向外部用户的稳定 import / CLI 用法
- SHOULD: 将以下情况视为辅助信号：
  - 根目录存在 `dist/`
  - 仓库内存在 wheel、sdist、构建脚本或其他典型发布产物
  - 项目名与 `src/` 下包目录明显对应
- MUST: 只有在存在至少一类强信号，或用户已明确说明当前仓库是对外发布包时，才切换为更严格的兼容性优先模式。

### 2. 兼容性优先级

- MUST: 默认优先保持公共 import 路径稳定。
- MUST: 默认优先保持模块导出、函数签名、参数语义、返回结构与异常行为稳定。
- MUST: 默认优先保持 CLI 参数、子命令、退出码语义与帮助信息中的关键用法稳定。
- MUST: 默认优先保持配置 key、配置文件结构、序列化字段、文件格式与默认行为稳定。
- MUST: 不要在未明确告知用户的情况下引入破坏性重命名、删除、重排、收紧类型约束或改变默认值语义。
- SHOULD: 将“用户现有代码是否还能运行”视为高于“内部实现是否更整洁”的约束。

### 3. 实施策略

- SHOULD: 优先做增量兼容修改，而不是直接替换旧接口。
- SHOULD: 若必须重构，优先保留旧别名、兼容层、re-export、参数兜底或弃用过渡。
- SHOULD: 修改模块布局或 `src/` 包结构前，先检查公开导入路径是否会失效。
- SHOULD: 调整函数参数时，优先新增可选参数，而不是直接删除、改名或重排现有参数。
- SHOULD: 调整返回值时，优先保持旧结构可用；若必须扩展，尽量采用向后兼容方式。
- SHOULD: 升级依赖时，同时检查 Python 版本下限、第三方依赖版本范围、安装方式与用户环境影响。
- MAY: 当旧行为确实需要淘汰时，先引入弃用提示，再在明确版本边界下移除。

### 4. 验证要求

- MUST: 修改前先识别哪些符号、导出模块、命令、配置项或文件格式已经对外暴露。
- MUST: 修改后至少检查以下内容是否仍一致或具备兼容过渡：
  - `README.md` 中的公开用法
  - 示例代码或文档中的 import 与调用方式
  - 测试中的公开接口覆盖
  - 包导出入口与 CLI 入口
- SHOULD: 若任务涉及 `pyproject.toml`、依赖、入口点或构建配置，额外检查发布与安装路径是否仍成立。
- MUST: 若存在潜在 breaking change，必须在结果汇报中明确指出影响面、迁移方式与是否已有兼容层。

### 5. 结果汇报

- MUST: 明确说明识别到了哪些“对外发布包”信号。
- MUST: 明确说明本次修改影响了哪些公开接口边界。
- MUST: 若保留了兼容层、旧别名或过渡方案，汇报中必须说明保留方式。
- MUST: 若无法避免破坏性变化，必须明确提示用户，而不是默认替用户接受 breaking change。

## 交付检查清单

在完成任务前，检查：

- 是否已识别到足以进入兼容性优先模式的发布信号
- 是否已盘点公共接口边界
- 是否避免了不必要的 breaking change
- 是否为兼容层、旧行为或过渡路径保留了验证
- 是否检查了 README、示例、测试或导出入口的一致性
- 是否在无法避免破坏时明确提示用户
