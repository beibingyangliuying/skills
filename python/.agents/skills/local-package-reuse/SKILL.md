# Local Package Reuse

## 核心目标

优先复用项目根目录中的本地包最新版候选，避免重复实现；当代码暂时无法直接回收到本地包时，先通过 `<本地包>_local` 过渡包收口实现，降低后续迁移成本。

## 执行规则

### 1. 本地包识别前置

- MUST: 在开始设计方案、搜索实现或编写代码前，先按 `local-package-discovery` 的规则检查当前项目根目录下是否存在本地 `.whl` 或 `.tar.gz` 发布产物。
- MUST: 将 discovery 阶段选出的最新版候选视为默认复用入口，而不是重新各自发明一套识别规则。
- SHOULD: 在决定引入公开第三方依赖、复制现有逻辑或新写公共代码前，先判断本地包最新版是否已经覆盖当前需求。
- MUST: 若 discovery 阶段未发现可用本地包，不要虚构本地包；应继续常规实现并如实说明未发现候选产物。

### 2. 复用提示与方案判断

- MUST: 若搜索、阅读或编写过程中发现现有代码的命名、职责、接口或业务语义与某个本地包明显对齐，立即提示“这里可能应优先复用本地包的最新版”，并指出对应产物与理由。
- SHOULD: 提示以建议复用为主，不强制中断当前任务，但默认决策应偏向优先评估本地包最新版。
- SHOULD: 若仅凭文件名无法判断本地包用途，应继续沿用 discovery 阶段的元数据与代码线索，而不是武断下结论。
- MUST: 若最终没有采用本地包方案，不要假装已经完成评估；只有在确实检查过相关产物或代码后，才能说明不采用的原因。

### 3. `<本地包>_local` 过渡包

- MUST: 若正在编写的代码明显属于某个本地包的能力边界，但当前又需要先在项目内落地，就在项目根目录创建 `<本地包>_local/` 包目录，而不是把逻辑直接散落到业务模块中。
- MUST: `<本地包>_local/` 作为本地包的过渡层与稳定依赖入口，承载临时实现、适配器、补丁逻辑或待回迁代码。
- MUST: `<本地包>_local/` 的名称使用规范化后的包名；若分发名包含 `-`，先转换为 `_`，再追加 `_local`。
- SHOULD: 新业务代码优先依赖 `<本地包>_local/`，减少未来升级本地包时的改动扩散。
- SHOULD: 当本地包升级并覆盖相同能力后，优先替换 `<本地包>_local/` 内部实现，再逐步回收调用点。
- MUST: 若存在多个候选本地包，应按语义最匹配的最新版决定归属，并创建对应的 `<本地包>_local/`。

## 交付检查清单

在完成任务前，检查：

- 是否已先按 `local-package-discovery` 的规则检查项目根目录中的 `.whl` / `.tar.gz`
- 是否已按分发名识别并优先考虑本地包最新版
- 是否在命中相符代码时主动提示优先复用本地包
- 若代码更适合沉淀到本地包，是否已改为放入 `<本地包>_local/`
- 若未采用本地包或 `_local` 方案，是否如实说明原因
