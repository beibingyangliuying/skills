# Design Review Notes

## 核心目标

把设计评审中那些“现在不一定立刻修改、但后续很值得持续关注”的问题，整理成一份轻量、可回看、可继续接上的 Markdown 备注，而不是让这些判断只停留在当前上下文里。

## 边界说明

- MUST: 本 skill 只负责设计层面的审查留档，不替代 `repo-workflow`、`python-code-style`、`dead-code-cleanup`、`python-test-organization` 等专项 skill。
- MUST: 本 skill 的默认目标是记录设计 concern，而不是立即修改代码；除非用户明确要求，不要把设计审查自动扩展为重构或修复任务。
- MUST: 仅记录会影响后续维护、扩展、重构、测试边界、模块职责、依赖方向或实现演进的问题，不记录局部格式、命名、import 顺序等轻量实现噪音。
- MUST NOT: 不得为了显得“全面”而机械罗列泛泛的架构套话，例如“耦合较高”“扩展性不足”“建议分层”，除非这些结论有明确代码依据。
- MUST NOT: 不得把未经证实的猜测、个人风格偏好或弱信号包装成确定性设计问题。
- SHOULD: 若任务重点已经变成“立刻修复问题”或“直接重构实现”，优先切换到更贴合场景的其他 skill，而不是继续用本 skill 承载实现动作。
- SHOULD: 若评审结果中包含长期有效、会反复影响后续决策的仓库隐式约定，可再按需衔接 `local-memory-capture`；但不要把设计备注和本地记忆机械合并。

## 主流程

### 1. 先确认任务是否属于“设计审查留档”

- MUST: 只有当用户明确希望评估设计、指出潜在问题、做后续提醒、形成审查备注或保留技术债观察时，才使用本 skill。
- MUST: 若任务只是一般 code review、bug 排查、测试修复或样式整理，不要误触发本 skill。
- SHOULD: 当用户表达类似“先不用改”“先记下来”“做个提醒”“留个备注”“看看以后可能出什么问题”时，优先按本 skill 处理。

### 2. 先限定评审范围

- MUST: 在输出设计备注前，先明确本次评审针对的范围，例如某个模块、某条调用链、某个子系统、某类抽象边界，或整个项目的实现组织方式。
- MUST: 若实际只看了局部代码，不要把结论表述成覆盖整个项目的全面判断。
- SHOULD: 在备注开头简要说明本次评审的背景、范围和为什么会做这次审查，避免后续回看时失去上下文。

### 3. 只记录高价值 concern

- MUST: 每条 concern 都应同时回答三个问题：
  - 观察到了什么现象
  - 为什么这件事后续可能会成为问题
  - 后续可以朝什么方向收敛
- MUST: 每条 concern 都要基于实际代码结构、模块关系、依赖流向、状态组织、配置方式、接口边界或测试组织给出依据。
- MUST: 若某条判断暂时证据不足，应明确写成“观察项”或“watchlist”，而不是上升为确定性问题。
- SHOULD: 优先记录以下几类问题：
  - 模块职责混杂或边界漂移
  - 依赖方向不自然或跨层直接耦合
  - 抽象泄漏，导致上层感知过多底层细节
  - 配置、状态或数据流入口扩散
  - 扩展点不稳定，后续新增能力时可能频繁改动旧代码
  - 设计选择已经开始影响测试边界、替换成本或演进难度
- SHOULD: 不必追求数量；宁可只记录 2–4 条真正重要的 concern，也不要堆一长串轻飘问题。

### 4. 生成轻量 Markdown 备注

- MUST: 输出应为轻量 Markdown 备注，而不是正式 issue 列表、风险台账系统或字段过重的审计报告。
- MUST: 默认使用叙述式结构，让人能快速读懂，不要求每条都附带编号、负责人、截止时间或优先级。
- MUST: 至少区分“本次主要 concern”和“仅供后续留意的 watchlist”。
- SHOULD: 备注开头在标题后补一行“评审时间：<填写时间>”，用于保留这次判断的时间上下文。
- SHOULD: 备注结构尽量简洁，默认包含：
  - Context
  - Main concerns
  - Watchlist
  - Not in scope
- SHOULD: 单条 concern 优先使用“小标题 + 2~4 句说明”的轻量写法，而不是表格。
- SHOULD: 默认采用类似下面的形态：

```md
# Design review notes

评审时间：<填写时间>

## Context
- Scope: <本次评审范围>
- Why this review was done: <为什么会做这次评审>

## Main concerns

### <问题标题>
<观察到的现象。>
<为什么它后续可能成为问题。>
<后续可考虑的收敛方向。>

### <问题标题>
<观察到的现象。>
<为什么它后续可能成为问题。>
<后续可考虑的收敛方向。>

## Watchlist
- <现在还不构成问题，但后续改动时值得留意的点>
- <另一个观察项>

## Not in scope
- <这次没有检查的部分，避免读者误以为已覆盖>
```
