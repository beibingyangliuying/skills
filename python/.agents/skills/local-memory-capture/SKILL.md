# Local Memory Capture

## 核心目标

让 AI 在规划与实现过程中，主动识别那些“单看代码本身不容易知道、但会显著影响后续决策或实现正确性”的隐性知识，并将其沉淀到本地记忆系统，而不是只停留在当前上下文里。

## 边界说明

- MUST: 本 skill 只约束目标项目中本地记忆的识别、查阅、沉淀与更新，不替代 `repo-workflow`、`python-code-style`、`python-test-organization` 等专项 skill。
- MUST: 只有当信息不能稳定地从当前代码、类型、测试、配置或常规仓库搜索直接得出时，才优先考虑写入本地记忆。
- MUST: 本地记忆只记录会影响未来规划、实现、验证或协作判断的项目级知识，不记录一次性调试噪音、偶发环境脏状态或泛泛而谈的通用建议。
- MUST NOT: 不得把尚未确认的猜测、弱推断或未经验证的解释写成记忆规则。
- MUST NOT: 不得机械抄写代码里已经直接可见的事实，例如函数返回值、模块路径、类层级或显式 schema。
- MUST: 默认从目标项目根目录下的 `doc/ai-memory/` 查找本地记忆；若找不到且无法从仓库结构推断替代位置，必须先询问用户。
- SHOULD: 若任务命中了其他专项 skill，本 skill 只补充“记忆查阅与沉淀”这一层，不扩大为主导实现规范。
- SEE: 需要判断“什么值得记录”、如何避免重复条目或如何写出低噪音记忆时，读取 `references/what-to-record.md`。

## 主流程

### 1. 先判断是否存在隐性知识风险

- MUST: 当任务涉及规划、方案比较、仓库约定确认、踩坑复盘、外部依赖认知、流程约束确认，或明确要求查阅 / 更新 `doc/` 下记忆时，优先判断是否存在隐性知识风险。
- SHOULD: 若任务明显依赖外部输入、历史约定、隐藏流程或仓库级 convention，在进入详细设计或编码前先检查本地记忆系统。

### 2. 在规划阶段先查本地记忆

- MUST: 默认先在目标项目中查找 `doc/ai-memory/`。
- MUST: 若目标项目中存在 `doc/ai-memory/README.md`，先查看索引，再按需读取最相关的记忆条目。
- MUST: 若 `doc/ai-memory/` 不存在、索引缺失、或项目明显使用其他知识目录命名，先向用户确认记忆目录位置，而不是直接创建。
- MUST: 若当前方案依赖某个尚未确认的外部事实，应先暴露该风险，而不是把它直接当成既定前提。
- SHOULD: 若用户在需求澄清或任务推进中补充了高价值、可复用的隐藏约束，可先标记为候选记忆，待信息确认后再写入。

### 3. 在实现阶段持续识别候选坑点

- MUST: 在编码、调试、验证过程中，一旦发现“这不是代码本身能可靠告诉你的规则”，立即触发记忆判定。
- MUST: 对以下四类信息默认视为高优先级候选：
- 需要外部输入才能知道的信息，单看代码推不出来。
- 仓库、团队、依赖库的隐式 convention，不提醒就容易做错。
- 环境或流程限制，例如必须走某个入口、某个命令顺序、某种生成或提交方式。
- 已被验证过、以后高概率重复踩到的失败模式或负面规则。
- SHOULD: 若已有相近记忆条目，优先更新现有条目，而不是新增重复文件。

### 4. 仅在确认后写入本地记忆

- MUST: 只有当信息已经被用户明确说明、被现有文档 / 流程确认、被实际验证失败印证，或被多次任务复用证明有稳定价值时，才写入本地记忆。
- MUST: 若目标项目采用 `doc/ai-memory/README.md` 作为索引，写入记忆时同步维护该索引；若项目采用其他等价索引文件，则维护项目实际使用的索引。
- MUST: 单条记忆文件使用独立的 Markdown 文件，frontmatter 至少包含 `name` 和 `description`。
- SHOULD: 单条记忆优先使用轻量模板，正文保持可扫描、可复用：

```md
---
name: <memory-name>
description: <什么时候应该读这条记忆，以及它约束什么>
---

## 适用场景

## 规则 / 坑点

## 证据或来源

## 失效条件
```

- SHOULD: 若目标项目采用 `doc/ai-memory/README.md`，该文件只做索引与导航，不承载长篇规则全文。

### 5. 失效时更新，而不是静默堆积

- MUST: 若旧记忆已与代码现状、现行流程或正式文档冲突，应更新条目并注明失效条件，而不是默默保留过期规则。
- SHOULD: 若某条记忆后来被更正式的项目文档覆盖，可将该条记忆收敛为短摘要或链接，避免并行维护两套完整说明。

## 交付检查清单

在完成任务前，检查：

- 是否先判断了当前任务是否存在隐性知识风险
- 是否已默认从目标项目的 `doc/ai-memory/` 查找本地记忆；若找不到，是否先询问了用户
- 若本地记忆系统存在，是否先查看了索引与相关条目
- 是否只记录了代码本身不易推出、但会影响后续工作的项目级知识
- 是否避免把猜测、一次性异常或代码可见事实写进记忆
- 若发现新坑点，是否先判断其是否满足“值得记录”的标准
- 新增或更新记忆条目时，是否同步维护了项目实际使用的索引
- 单条记忆是否保持独立、轻量、可按需加载，而不是把索引文件变成大而全手册
