# What To Record

本文件细化 `local-memory-capture` 中“什么值得写入本地记忆”的判断标准，以及新增 / 更新条目时的写法约束。

## 查找与创建边界

- MUST: 记录标准和目录存在性是两回事；本 skill 负责约束“什么值得记、怎么查、怎么写”，不假定目标项目一定已经初始化本地记忆目录。
- MUST: 默认先在目标项目根目录下查找 `doc/memories/`。
- MUST: 若 `doc/memories/README.md` 存在，先读索引，再按需读取相关条目。
- MUST: 若 `doc/memories/` 不存在、索引缺失、或仓库明显使用其他等价知识目录，先询问用户确认位置，而不是直接创建或强行迁移。
- SHOULD: 若用户尚未指定位置，才将 `doc/memories/` 作为推荐默认值提出。
- SHOULD: 若用户在具体项目中采用其他语义等价的位置或索引文件，应遵循用户指定位置。

## 优先记录的四类信息

### 1. 外部输入依赖

- MUST: 当某项判断依赖用户口头说明、团队口径、外部系统现状、人工流程、历史背景或其他仓库外信息，且这些信息不会稳定地体现在代码里时，优先视为候选记忆。
- SHOULD: 若没有这条外部输入，合理的 AI 很可能会做出另一个同样“看起来合理”但实际错误的决定，则更应记录。

典型例子：

- 某个入口虽然存在，但项目规定只能通过包装脚本调用。
- 某个环境变量或配置值由外部平台注入，仓库中不会直接出现。

### 2. 隐式 convention

- MUST: 若仓库、团队或依赖库在本项目里有“默认做法”，而这个做法不是官方默认、也不是从局部代码一眼可见的，就应作为候选记忆评估。
- MUST: 对隐式 convention 的判断标准是：如果没人提醒，合理的 AI 很可能做错。
- SHOULD: 优先写成可执行的约束句，而不是长篇背景故事。

典型例子：

- 调用某个库时必须经过本项目的适配层，而不是直接调用官方入口。
- 生成、发布或同步流程虽然能手动执行多个入口，但项目约定只允许其中一个标准路径。

### 3. 环境或流程限制

- MUST: 若某个任务只有按固定入口、固定顺序、固定环境或固定审批方式执行才可靠，应作为候选记忆评估。
- SHOULD: 如果错误顺序会导致结果错误、产物不一致或难以排查的问题，应优先沉淀。

典型例子：

- 必须先跑 A 再跑 B，否则生成结果不正确。
- 某类命令必须在特定 shell、特定虚拟环境或特定工作目录中执行。

### 4. 已验证的失败模式

- MUST: 只有当失败模式已经被用户纠正、被验证失败直接印证、或已从现有文档 / 流程中确认时，才允许记录。
- MUST: 对失败模式的记录重点应是“如何避免再次踩坑”，而不是堆砌问题复盘细节。
- SHOULD: 若未来任务高概率再次触发同类误用，应优先记录为负面规则。

典型例子：

- 某个字段顺序看似自由，实际一旦交换就会破坏下游兼容性。
- 某个工具默认参数在本项目中会产生错误输出，因此必须显式关闭或替换。

## 不应记录的内容

- MUST NOT: 不得记录代码、测试、配置、schema 或标准仓库搜索已能直接确认的事实。
- MUST NOT: 不得记录一次性网络波动、临时缓存异常、偶发本地脏状态等低复用噪音。
- MUST NOT: 不得记录尚未确认的猜测、印象式总结或“可能是这样”的推断。
- MUST NOT: 不得把泛泛而谈的工程常识写成本项目的本地记忆。
- SHOULD NOT: 若已有相近条目且本次没有新增约束、适用边界或失效条件，不要重复新建文件。

## 记录前的最小判断

在新增或更新本地记忆前，至少确认以下问题：

1. 这条信息是否不能稳定地从当前代码或常规搜索直接得出？
2. 如果没有这条提醒，后续 AI 是否很容易做出错误但表面合理的选择？
3. 这条信息是否已经被用户、文档、流程或实际验证充分确认？
4. 这条信息是否值得在未来任务中复用，而不只是当前一次性上下文？

若以上问题中有任一答案明显是否定的，默认先不要写入。

## 写法约束

- MUST: 每条记忆使用独立文件，frontmatter 至少包含 `name` 与 `description`。
- MUST: 文件名使用短横线命名，并直接体现主题。
- MUST: 正文优先描述规则、坑点、适用边界与失效条件，不要堆砌无关背景。
- MUST: 若目标项目采用 `doc/memories/README.md` 作为索引，则同步维护该索引；若项目使用其他等价索引文件，则维护实际使用的索引。
- SHOULD: 优先使用短句和清晰标题，让后续任务可以按需快速加载。
- SHOULD: 若同一主题已经存在条目，优先补充“适用场景”“规则 / 坑点”“失效条件”，不要平行创建近义条目。
- SHOULD: 若项目后来已有正式文档覆盖该知识，可把记忆条目收敛为摘要和链接，降低维护成本。

## 推荐模板

```md
---
name: release-workflow-constraints
description: 当任务涉及发布、构建或生成流程时读取，说明必须遵守的入口与顺序约束。
---

## 适用场景

- 发布、构建、代码生成或同步任务。

## 规则 / 坑点

- 必须先执行步骤 A，再执行步骤 B。
- 不要直接调用官方入口，必须通过项目包装脚本。

## 证据或来源

- 用户在 2026-04-14 的任务说明中明确要求。
- 已在实际验证中观察到反向顺序会产生错误结果。

## 失效条件

- 若后续正式文档明确替代该流程，或项目已移除包装脚本约束，应同步更新此条目。
```
