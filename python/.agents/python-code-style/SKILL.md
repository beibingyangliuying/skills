# Python Code Style

## 何时使用

当任务包含以下任一情况时使用本 skill：

- 新增或重构 Python 函数
- 修改类定义、方法签名或模块接口
- 抽取公共逻辑
- 补充或修正文档字符串
- 添加或整理类型注解

## 核心目标

在遵循仓库既有风格的前提下，生成可读、可维护、低副作用的 Python 代码。

## 执行规则

### 1. 函数与接口设计

- MUST: 函数名、参数名、返回值语义必须清晰。
- MUST: 公共函数的参数设计应稳定、直接，避免无必要的位置参数陷阱。
- SHOULD: 优先写短函数；单个函数应聚焦单一职责。
- SHOULD: 公共逻辑应提取为具名函数，而不是堆叠在调用处。
- SHOULD: 对复杂输入约束、返回结构、边界条件给出必要文档。
- MAY: 对只在局部使用且逻辑极小的辅助函数保持简洁。

### 2. 类型注解

- MUST: 使用 `|` 表达联合类型与可空类型。
- MUST: 不使用已弃用的 `typing` 别名。
- SHOULD: 公共函数显式标注返回类型。
- SHOULD: 参数类型尽量具体，但不要为了“看起来严格”而引入不必要的复杂泛型。
- SHOULD: 简单局部变量通常不写类型注解。
- MAY: 对复杂中间对象补充局部类型注解以帮助理解。

### 3. 文档字符串

- SHOULD: 文档字符串使用 Google Style。
- SHOULD: 文档只写必要信息，避免重复函数签名中已清楚表达的内容。
- MUST: 当输入或输出包含复杂结构约束时，文档中必须说明。
- SHOULD: 说明以下内容中的必要部分：
  - 参数含义
  - 返回值结构
  - 关键前置条件
  - 重要副作用
  - 异常或错误条件

### 4. 编程范式

- SHOULD: 优先采用低副作用实现。
- SHOULD: 能直接表达意图时，优先简单、直接的函数式写法。
- SHOULD: 仅在确有收益时引入类，不为形式而包装成类。

### 5. 数据模型

- SHOULD: 若仓库中的轻量数据模型主要采用 `attrs`，则新增同类模型时优先沿用 `attrs`。
- SHOULD: 使用 `attrs` 时优先考虑不可变设计，例如 `frozen=True`，但前提是不妨碍当前任务。
- MUST: 不要为了使用 `attrs` 而破坏仓库既有范式；若目标文件已明确采用其他模式，应优先保持一致。

## 交付检查清单

在完成代码前，检查：

- 接口命名是否清晰
- 返回类型是否需要显式标注
- 文档是否覆盖复杂结构约束
- 是否引入了不必要的副作用
- 是否保持了目标文件的既有风格
